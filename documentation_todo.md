# Documentation TODO

Below is every source file that needs to be looked at and documentation generated for. Work through them **one by one**: read the file, write (or extend) the matching docs page in `quidproquo-docusaurus`, then tick the checkbox.

## What we care about

For each file, we only care about two kinds of exports — everything else (internal helpers, plumbing types) can be skipped or folded into the page of the export it supports:

1. **Config settings** — the `defineXYZ(...)` functions (e.g. `defineStorageDrive`). These declare infrastructure/config.
2. **Action requesters** — the `askXYZ(...)` generator functions (e.g. `askFileReadTextContents`). These are what stories yield.

If a file contains neither (it's just types, enums, utils), tick it off and document its types on the page of the define/ask function that uses them.

**Only document the public API.** If a define/ask function isn't exported from its package's top-level index (i.e. a user can't import it from `quidproquo-core`/`quidproquo-webserver`/etc.), it's internal — tick it off without a page.

**Expect to tick off far more files than you write pages for.** Whole packages in this list (`quidproquo-actionprocessor-*`, `quidproquo-deploy-awscdk`, `quidproquo-dev-server`, `quidproquo-testing`) contain almost no define/ask exports — they're the reference material you read to understand what things do, not documentation targets themselves. Suggested order: `quidproquo-core` config → core actions → `quidproquo-webserver` → the rest.

## Where the docs go

The docusaurus site lives in `quidproquo-docusaurus`. Two sections exist under `docs/`, each broken up by project (core, webserver, features, etc.):

- **Config** → `docs/config/<project>/<setting-name>.md` — one page per define function.
- **Action Requesters** → `docs/actions/<project>/<domain>/<ask-function-name>.md` — one page per action requester, grouped by action domain (file, queue, event-bus, ...).

`<project>` is the short package name: `core`, `webserver`, `features`, `xstate`, etc. The sidebar is **autogenerated** from these directories (see `sidebars.ts`), so new pages appear automatically — just add a `_category_.json` when you create a new folder (copy an existing one for the format).

**File naming:** kebab-case. Config pages drop the `define` prefix (`defineStorageDrive` → `storage-drive.md`); action pages keep the `ask` prefix (`askFileReadTextContents` → `ask-file-read-text-contents.md`). The page `title:` front-matter is always the exact function name.

## Worked examples — match these exactly

Three example pages exist; use them as the template for structure, tone, front-matter, and cross-linking:

- Config: `quidproquo-docusaurus/docs/config/core/storage-drive.md` (defineStorageDrive)
- Actions: `quidproquo-docusaurus/docs/actions/core/file/ask-file-read-text-contents.md` and `ask-file-write-text-contents.md`

## Rules for writing a page

- **Types are super important.** Explain every argument, and every property of options objects — including nested types (enums, unions, sub-interfaces) with defaults and what each value means. Use tables like the examples.
- **Work out what things are actually for.** If the purpose of a setting/action isn't obvious from quidproquo-core, look at how it's consumed in `quidproquo-deploy-awscdk` (what infrastructure it creates) and the `quidproquo-actionprocessor-*` packages (how it executes at runtime), and say so on the page (e.g. "On AWS: deploys an S3 bucket").
- **Cross-link related pages.** Actions should link to the config page that controls them and to related actions; config pages should link back to the actions that use them. Example: a file read action links to the storage drive config page. Use relative `.md` links (`../../../config/core/storage-drive.md`) — the site's routeBasePath is `/`, so absolute `/docs/...` links break.
- **Links are two-way.** Whenever you add a link from page A to page B, open page B and add a link back to page A in its Related section (if it isn't there already). Related pages written earlier in this list won't know about pages written later — updating the other side as you go is what keeps the Related sections complete. If page B doesn't exist yet, mention the function by name in prose (no link) and add the real link when B's page gets written.
- Document errors (`createErrorEnumForAction` enums), return types, and include at least one realistic story-code example per page. Examples are always usage **inside a story** — a generator function composing the action with `yield*` — never a bare call:

  ```typescript
  export function* askSaveReport(reportCsv: string) {
    yield* askFileWriteTextContents('reports', 'exports/monthly.csv', reportCsv);
  }
  ```
- **Write from the consumer's perspective, not the repo's.** Don't put source file paths on pages — users import from the package (`quidproquo-core`), the repo layout is irrelevant to them. When referencing values, prefer the exported enum member over its underlying string: `FileActionType.ReadTextContents`, not `@quidproquo-core/File/ReadTextContents`.
- **Action requester signatures return `AskResponse<T>`.** Don't show the internal `...ActionRequester` alias in signatures — resolve it to what the story actually receives. The `T` is the second generic argument of the `ActionRequester<Action, T>` type in the file's `...ActionTypes.ts`, e.g. `FileWriteTextContentsActionRequester = ActionRequester<FileWriteTextContentsAction, void>` documents as `): AskResponse<void>;`.
- **Real-world usage examples:** `~/repo/antero-software/doccypoccy` is a real application built on quidproquo (see its `apps/` and `packages/`). Grep it for the define/ask function you're documenting — it often has actual in-flow usage (stories composing several actions, real config) that makes a far better example than a bare function call. Adapt what you find rather than copying verbatim (rename app-specific things), and don't reference doccypoccy by name in the published docs.
- The old domain-level pages under `docs/api/actions/` are the legacy overview — don't extend them; new per-function pages go in the sections above.
- After a batch of pages, run `npm run build` in `quidproquo-docusaurus` — the build fails on broken links.
- Tick the file's checkbox below (`[ ]` → `[x]`) once its page is written or it's been folded into another page.
